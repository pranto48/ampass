# AMPass — Security Review & Production Readiness

## WARNING

AMPass requires professional security audit before real credential storage. This document outlines the threat model and hardening checklist but does NOT constitute a security certification.

## Threat Model

### Assets Protected
- User vault credentials (passwords, notes, card numbers, SSH keys)
- Master password (never stored)
- Vault encryption key (memory-only, derived from master password)
- Auth tokens (OS keychain on desktop, session on web)
- Backup encryption passwords
- Remote backup credentials (FTP/SFTP/OneDrive)

### Threat Actors
1. **External attacker** — network access, no server access
2. **Compromised server** — attacker has read access to database/files
3. **Compromised client** — malware on user's device
4. **Insider/admin** — server admin with full DB access
5. **Supply chain** — compromised update package

### Security Boundaries
- **Client-side encryption**: Server never sees plaintext vault data
- **Zero-knowledge**: Even with full DB access, vault items cannot be decrypted without user's master password
- **Token-based auth**: Extension/desktop use bearer tokens, not session cookies
- **Encrypted at rest**: Backup configs, API keys, tokens encrypted with APP_SECRET

## Production Hardening Checklist

### HTTPS (Critical)
- [ ] Enable HTTPS on production server
- [ ] Uncomment HSTS header in .htaccess
- [ ] Uncomment HTTP→HTTPS redirect in .htaccess
- [ ] Verify extension API rejects non-HTTPS (except localhost)
- [ ] Set `session.cookie_secure = 1` in PHP

### APP_SECRET (Critical)
- [ ] APP_SECRET is a random 64-character hex string (generated by installer)
- [ ] APP_SECRET is never committed to version control
- [ ] APP_SECRET is never logged or exposed in error messages
- [ ] Changing APP_SECRET invalidates all encrypted configs (remote backup, email, GitHub token)

### File Permissions (Critical)
- [ ] `config/config.php` — 640 or 600 (owner read only)
- [ ] `config/.install_lock` — exists and not writable by web
- [ ] `app_storage/` — 750 (not world-readable)
- [ ] `app_storage/backups/` — 750
- [ ] `app_storage/releases/` — 750
- [ ] `.htaccess` blocks direct access to: config/, app/, database/, docs/, app_storage/
- [ ] `install/` directory deleted after installation

### Database Security
- [ ] MySQL user has minimal privileges (no GRANT, no FILE, no PROCESS)
- [ ] Database password is strong and unique
- [ ] Remote MySQL access disabled unless required
- [ ] All queries use prepared statements (no string interpolation)

### Session Security
- [ ] Session cookie: HttpOnly, Secure (HTTPS), SameSite=Strict
- [ ] Session regenerated on login
- [ ] Session fingerprinting (IP + User-Agent)
- [ ] Vault unlock state cleared on session destroy
- [ ] Idle timeout configured (default 5 min for vault)

### Extension/Desktop API
- [ ] CORS restricted to configured extension origins only
- [ ] Bearer tokens have configurable expiry (default 30 days)
- [ ] Token revocation works (admin can revoke devices)
- [ ] Rate limiting on all API endpoints
- [ ] HTTPS required for all authenticated endpoints (except localhost dev)

### Updater Security
- [ ] Update source configured (release or branch)
- [ ] GitHub token encrypted at rest (if used for private repos)
- [ ] ZIP entries validated before extraction (no path traversal)
- [ ] Rollback on failure (overwritten files restored, new files deleted)
- [ ] Failed migrations never marked as applied
- [ ] Maintenance mode during update
- [ ] Pre-update encrypted backup created automatically
- [ ] Download SHA-256 logged for audit
- [ ] **TODO**: Signed release verification for production (GPG or checksum file)

### Backup Security
- [ ] Backups encrypted with user-provided password (AES-256-GCM or XChaCha20)
- [ ] Backup password never stored on server
- [ ] Remote backup configs encrypted with APP_SECRET (never plaintext base64)
- [ ] Only .ampass-backup files uploaded remotely
- [ ] FTP/FTPS size verification after upload
- [ ] OneDrive refresh token encrypted at rest
- [ ] SFTP preferred over FTPS (stronger TLS verification)

### Input Validation
- [ ] All user input sanitized (Security::sanitize)
- [ ] Email validated (Security::isValidEmail)
- [ ] File uploads: strict extension allowlist + type mapping
- [ ] Stored filenames randomized (no user-controlled paths)
- [ ] Path traversal protection on all file operations
- [ ] ZIP extraction validates every entry before extracting

### Output Security
- [ ] All output HTML-escaped (htmlspecialchars)
- [ ] CSP header configured
- [ ] X-Content-Type-Options: nosniff
- [ ] X-Frame-Options: DENY
- [ ] No sensitive data in error messages shown to users
- [ ] Audit logs never contain plaintext passwords

### Rate Limiting
- [ ] Login: 5 attempts per 15 minutes per IP
- [ ] Register: 3 attempts per hour per IP
- [ ] Vault unlock: 5 attempts per 5 minutes per user+IP
- [ ] Extension login: 5 attempts per 15 minutes per IP
- [ ] API endpoints: configurable per-endpoint limits

### Audit Logging
- [ ] All auth events logged (login, logout, failed attempts)
- [ ] All vault operations logged (create, update, delete)
- [ ] All admin actions logged (settings, user management, updates)
- [ ] Extension actions logged separately (autofill, autosave, device registration)
- [ ] Logs never contain plaintext secrets
- [ ] Log retention policy configured

## Known Limitations

### Not Yet Implemented
- GPG-signed release verification for updater
- SFTP host key fingerprint pinning (PHP ssh2 limitation)
- Hardware security key (WebAuthn/FIDO2) support
- Client-side password breach checking (HaveIBeenPwned k-anonymity)
- Automatic backup scheduling (requires cron)
- Multi-factor authentication (TOTP prepared but not active)

### Architecture Limitations
- PHP session-based web auth (standard for shared hosting)
- No end-to-end encrypted sharing (sharing uses re-encryption)
- FTPS TLS verification depends on PHP/OpenSSL build
- Desktop app trusts OS keychain security
- Browser extension trusts Chrome/Edge extension sandbox

### Deployment Considerations
- Shared hosting: Other sites on same server may access files if permissions are wrong
- XAMPP development: No HTTPS by default, relaxed security acceptable for local dev only
- cPanel: Ensure .htaccess is processed (AllowOverride All)
- Docker: Mount app_storage as persistent volume, set proper UID/GID

## Incident Response

If a security breach is suspected:
1. Enable maintenance mode immediately
2. Revoke all extension tokens (Admin → Extensions)
3. Force password reset for affected users
4. Create encrypted backup
5. Review audit logs for unauthorized access
6. Rotate APP_SECRET (invalidates all encrypted configs)
7. Regenerate all remote backup credentials
8. Notify affected users

## Contact

For security issues, contact the repository maintainer directly. Do not open public GitHub issues for security vulnerabilities.
