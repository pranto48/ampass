# AMPass Desktop App — Tauri v2 Specification

## 1. Value Proposition

The desktop app provides capabilities that the PWA and browser extension cannot:

| Feature | PWA | Extension | Desktop App |
|---------|-----|-----------|-------------|
| Offline vault access | ❌ | ❌ | ✅ Encrypted local cache |
| OS keychain integration | ❌ | ❌ | ✅ Windows Credential Manager |
| System tray with lock status | ❌ | ❌ | ✅ Native tray icon |
| Native file picker for backups | ❌ | ❌ | ✅ OS file dialog |
| Global keyboard shortcuts | ❌ | Limited | ✅ System-wide |
| Native messaging bridge | ❌ | Needs host | ✅ Acts as host |
| Auto-lock on OS idle/sleep | ❌ | ❌ | ✅ OS event listeners |
| Windows installer (.msi/.exe) | ❌ | ❌ | ✅ Tauri bundler |
| Persistent background process | ❌ | Limited | ✅ Tray app |

**Conclusion:** The desktop app is justified for users who want offline access, OS-level security integration, and a persistent tray app. It does NOT replace the web vault or extension.


### Key Flows

**Login:**
1. User enters server URL + credentials in desktop app
2. App calls `POST /api/extension/login` (same API as browser extension)
3. Receives bearer token + derivation params
4. Stores token encrypted in OS keychain via Rust `keyring` crate
5. Derivation params stored in encrypted local cache

**Unlock:**
1. User enters master password
2. Frontend derives vault key via PBKDF2 (Web Crypto API)
3. Vault key stored in memory only (Rust holds a reference)
4. Optionally: vault key encrypted with device key and cached for "remember device" feature

**Sync:**
1. On unlock, fetch full vault from server (`GET /api/extension/vault/list`)
2. Store encrypted items in local cache (already encrypted — just write ciphertext to disk)
3. On subsequent opens, load from cache first, then sync in background
4. Conflict resolution: server wins (last-write-wins for now)

**Offline Mode:**
1. If server unreachable, load from local encrypted cache
2. Show "offline" indicator
3. Read-only mode (no saves until reconnected)
4. When connection restored, re-sync automatically

## 5. Security Model

### What is stored on disk

| Data | Location | Protection |
|------|----------|------------|
| Bearer token | OS Keychain | Windows Credential Manager encryption |
| Device key (random 256-bit) | OS Keychain | Windows Credential Manager encryption |
| Encrypted vault cache | `%APPDATA%/ampass/cache.enc` | AES-256-GCM with device key |
| Server URL | `%APPDATA%/ampass/config.json` | Plaintext (non-sensitive) |
| User preferences | `%APPDATA%/ampass/config.json` | Plaintext (non-sensitive) |

### What is NEVER stored on disk

- Master password (used transiently for PBKDF2, then discarded)
- Vault key in plaintext (only in memory while unlocked)
- Decrypted vault item fields
- Plaintext passwords, card numbers, notes, or any secret

### Encryption layers

1. **Vault items**: Already encrypted by the user's vault key (AES-256-GCM). The desktop app stores them as-is (ciphertext).
2. **Local cache file**: The entire cache file is additionally encrypted with a device key stored in OS keychain. This prevents reading the cache if the keychain is not accessible.
3. **OS Keychain**: Windows Credential Manager provides OS-level protection (tied to Windows user account, encrypted at rest by Windows).

### Lock behavior

- **Auto-lock on idle**: Rust monitors system idle time. After configurable timeout (default 15 min), vault key is wiped from memory.
- **Auto-lock on sleep/lock**: OS event listener triggers immediate lock.
- **Manual lock**: Tray menu "Lock Vault" or keyboard shortcut.
- **Lock clears**: Vault key from memory. Cached items remain on disk (encrypted). User must re-enter master password to unlock.

### Wipe local cache

User can trigger "Wipe Local Data" from settings:
- Deletes `cache.enc`
- Removes keychain entries
- Resets to fresh state
- Does NOT affect server data

## 6. Sync Model with PHP Server

### Initial sync
1. After login + unlock, fetch all vault items from server
2. Write encrypted items to local cache (they're already ciphertext)
3. Store `last_sync_at` timestamp

### Incremental sync (future optimization)
- Server could add `updated_since` parameter to vault/list endpoint
- Desktop app sends `last_sync_at` to only fetch changed items
- For v1: full sync on every unlock (simple, correct)

### Conflict resolution
- **Server is authoritative** — if an item was modified on both server and local cache, server version wins
- Local edits are pushed to server immediately when online
- If offline, edits are queued and pushed on reconnect (v2 feature — v1 is read-only offline)

### Sync triggers
- On vault unlock
- Every 5 minutes while unlocked (background poll)
- On window focus (if >1 minute since last sync)
- Manual "Sync Now" button

## 7. Offline Mode Behavior

| State | Behavior |
|-------|----------|
| Online + Unlocked | Full read/write. Changes sync immediately. |
| Online + Locked | Tray shows locked icon. Must unlock to access. |
| Offline + Unlocked | Read-only from local cache. "Offline" badge shown. |
| Offline + Locked | Cannot unlock (no server to verify master password*). |

*Note: For offline unlock to work, the app would need to store the master password hash locally. This is a security tradeoff:
- **Option A (secure, v1)**: Require server for unlock. Offline = locked.
- **Option B (convenient, v2)**: Store Argon2id hash of master password locally (encrypted with device key). Allow offline unlock. Risk: if device key is compromised, attacker can brute-force the master password hash.

**v1 recommendation**: Option A. Offline mode only works if vault was already unlocked before going offline.

## 8. Native Messaging Bridge Plan

The desktop app can act as a Chrome Native Messaging Host, allowing the browser extension to communicate with it.

### Use case
- Extension detects desktop app is running
- Instead of asking user for master password, extension requests vault key from desktop app
- Desktop app confirms via native notification ("Allow extension to access vault?")
- If approved, sends vault key to extension via native messaging
- Result: user unlocks once in desktop app, extension stays unlocked automatically

### Implementation

**Chrome native messaging manifest** (`com.ampass.desktop.json`):
```json
{
  "name": "com.ampass.desktop",
  "description": "AMPass Desktop App Native Messaging Host",
  "path": "C:\\Program Files\\AMPass\\ampass-native-host.exe",
  "type": "stdio",
  "allowed_origins": ["chrome-extension://YOUR_EXTENSION_ID/"]
}
```

**Protocol** (JSON over stdin/stdout):
```json
// Extension → Desktop
{ "type": "PING" }
{ "type": "REQUEST_VAULT_KEY" }
{ "type": "GET_STATUS" }

// Desktop → Extension
{ "type": "PONG", "version": "1.0" }
{ "type": "VAULT_KEY", "key": "hex..." }  // Only after user approval
{ "type": "STATUS", "unlocked": true }
{ "type": "DENIED" }
```

**Security**:
- Only allowed extension IDs can connect (manifest restriction)
- Desktop app shows native notification asking user to approve
- Vault key transfer happens over stdio (local, not network)
- If desktop app is locked, returns DENIED

### Registry entry (Windows)
```
HKCU\Software\Google\Chrome\NativeMessagingHosts\com.ampass.desktop
  (Default) = "C:\Program Files\AMPass\com.ampass.desktop.json"
```

   - Recommendation: Full sync for v1. Add `updated_since` to server API for v2.

## ⚠️ Security Warning

**This desktop application specification has NOT been professionally audited.**

The desktop app introduces additional attack surface compared to the web vault:
- Local encrypted cache could be targeted by malware with disk access
- OS keychain security depends on Windows user account protection
- Native messaging bridge creates a local IPC channel
- Process memory could be inspected by privileged processes

**Do NOT use for real credentials until professionally audited.**
